## 用户场景与测试

### 用户故事 1 - 配置自动初始化与数据持久化 (优先级: P1) 🎯 MVP

作为开发者，我希望在首次运行任何 `asw` 命令时自动创建配置目录和默认配置文件，并在后续操作中能够持久化保存配置数据，以便在不同会话间保留我的模型配置，而无需手动执行初始化命令。

**为什么是此优先级**: 这是整个系统的基础，没有配置存储和数据结构，所有其他功能都无法运行。它是实现 CLI 工具的核心前提。自动初始化符合现代 CLI 工具的使用习惯（如 git、npm），提供更好的用户体验。

**独立测试**: 可以通过首次执行任何 `asw` 命令（如 `asw model list`）来测试自动初始化行为，验证配置目录和文件的自动创建、默认内容的生成、以及重复执行时的幂等性。

**验收场景**:

1. **给定** 首次使用 AgentSwitch（配置目录不存在），**当** 执行任何 `asw` 命令（如 `asw model list`），**则** 在 `~/.agentswitch/` 路径自动创建配置目录和包含空模型列表的 `config.toml` 文件，然后继续执行用户请求的命令
2. **给定** 配置目录已存在，**当** 执行任何 `asw` 命令，**则** 跳过目录创建并直接执行用户请求的命令（幂等操作）
3. **给定** 配置目录创建失败（如权限不足），**当** 执行任何 `asw` 命令，**则** 输出包含具体失败原因的错误信息并以非零退出码退出
4. **给定** 合法的配置数据，**当** 保存到配置文件，**则** 能够重新读取并获取与原始数据完全一致的内容（往返一致性）

---

### 用户故事 2 - 模型配置管理命令 (优先级: P1) 🎯 MVP

作为开发者，我希望通过命令行添加、查看、编辑和删除模型配置，以便快速管理不同的 API 提供商配置。

**为什么是此优先级**: 这是用户与系统交互的主要方式，提供了配置管理的核心功能。没有这些命令，用户无法实际使用系统。

**独立测试**: 可以通过执行完整的 CRUD 操作流程测试：添加模型 → 列出模型 → 编辑模型 → 删除模型，每个步骤都有明确的输出和验证点。

**验收场景**:

1. **给定** 没有模型配置，**当** 执行 `asw model add <name> --base-url <url> --api-key <key> --model <id>` 命令，**则** 创建并保存模型配置，输出绿色成功信息
2. **给定** 存在模型配置，**当** 执行 `asw model list` 命令，**则** 以表格形式显示所有模型的 name、base_url、model_id 和掩码后的 api_key
3. **给定** 存在模型配置，**当** 执行 `asw model remove <name>` 命令，**则** 从配置中删除该模型并输出绿色成功信息
4. **给定** 存在模型配置，**当** 执行 `asw model edit <name> --api-key <new-key>` 命令，**则** 更新指定字段并输出绿色成功信息
5. **给定** 添加同名模型配置，**当** 执行 `asw model add` 命令，**则** 输出红色错误信息，提示该名称已被使用
6. **给定** 删除不存在的模型配置，**当** 执行 `asw model remove <name>` 命令，**则** 输出红色错误信息，提示未找到该配置
7. **给定** 没有任何模型配置，**当** 执行 `asw model list` 命令，**则** 输出黄色提示信息，建议用户使用 `asw model add` 命令

### 用户故事 3 - 安全性与用户体验增强 (优先级: P2)

作为开发者，我希望 CLI 输出有清晰的视觉区分，并且 API Key 得到安全处理，以便快速识别操作结果并保护敏感信息。

**为什么是此优先级**: 安全性和用户体验是重要的质量属性，但不阻塞核心功能的实现。在 MVP 完成后可以逐步完善。

**独立测试**: 可以通过验证错误场景、警告信息和 API Key 掩码显示来测试，确保所有输出都符合颜色和安全规范。

**验收场景**:

1. **给定** 列出模型配置，**当** 包含 API Key，**则** 仅显示前 4 个字符加 `****` 后缀
2. **给定** 保存配置文件，**当** 写入操作完成，**则** 配置文件权限设置为仅所有者可读写（Unix 权限 0600）
3. **给定** 操作成功，**当** 输出结果信息，**则** 使用绿色文本并带有 `✓` 前缀
4. **给定** 操作失败，**当** 输出错误信息，**则** 使用红色文本并带有 `✗` 前缀，包含具体错误原因
5. **给定** 输出警告信息，**当** 显示提示，**则** 使用黄色文本
6. **给定** 提供命令建议，**当** 显示信息性提示，**则** 使用蓝色文本

### 边界情况

- **URL 验证**: 当用户提供无效的 URL 格式作为 base_url 时，系统如何拒绝并提示？
- **空参数**: 当用户提供的 name 参数为空字符串时，系统如何提示并拒绝？
- **并发访问**: 当多个进程同时尝试读写配置文件时，系统如何处理文件锁？
- **配置文件损坏**: 当配置文件不是合法的 TOML 格式时，系统如何提供友好的错误信息？
- **磁盘空间不足**: 当保存配置时磁盘空间不足，系统如何处理？
- **权限问题**: 当配置目录不可读或不可写时，系统如何提供清晰的错误指导？
- **特殊字符**: 当模型名称或 API Key 包含特殊字符时，系统如何正确处理？
- **超长输入**: 当用户提供超长的 URL 或 API Key 时，系统如何验证和限制？

## 需求

### 功能性需求

#### 配置管理 (FR-001 至 FR-005)

- **FR-001**: 系统 MUST 在首次运行任何命令时自动检测并在 `~/.agentswitch/` 路径创建配置目录（如果不存在）
- **FR-002**: 系统 MUST 在配置目录中创建 `config.toml` 文件用于持久化存储（如果不存在）
- **FR-003**: 系统 MUST 在配置文件不存在时自动创建包含空模型列表和空活跃模型映射的默认配置
- **FR-004**: 系统 MUST 支持从配置文件读取配置数据
- **FR-005**: 系统 MUST 将配置数据以 TOML 格式写入配置文件

#### 数据结构 (FR-006 至 FR-009)

- **FR-006**: 系统 MUST 定义包含 name、base_url、api_key、model_id 字段的模型配置数据结构
- **FR-007**: 系统 MUST 支持在模型配置中存储额外的键值对参数（extra_params）
- **FR-008**: 系统 MUST 维护一个模型配置列表（AppConfig.models）
- **FR-009**: 系统 MUST 维护一个从 Agent 名称到模型名称的映射（AppConfig.active_models）

#### 模型配置管理命令 (FR-010 至 FR-017)

- **FR-010**: 用户 MUST 能够通过命令行添加新的模型配置（`asw model add`）
- **FR-011**: 用户 MUST 能够查看所有已配置的模型列表（`asw model list`）
- **FR-012**: 用户 MUST 能够删除指定的模型配置（`asw model remove`）
- **FR-013**: 用户 MUST 能够编辑已存在的模型配置（`asw model edit`）
- **FR-014**: 系统 MUST 验证模型名称不能为空字符串
- **FR-015**: 系统 MUST 验证 base_url 必须是合法的 URL 格式
- **FR-016**: 系统 MUST 拒绝添加同名的模型配置
- **FR-017**: 系统 MUST 在删除模型时清除对应的活跃模型映射

#### 序列化与数据完整性 (FR-018 至 FR-020)

- **FR-018**: 系统 MUST 将配置序列化为合法的 TOML 格式
- **FR-019**: 系统 MUST 从合法的 TOML 格式反序列化配置
- **FR-020**: 系统 MUST 保证序列化后再反序列化的往返一致性（所有字段值与原始对象一致）

#### 安全性 (FR-021 至 FR-023)

- **FR-021**: 系统 MUST 在列出模型配置时对 API Key 进行掩码处理（仅显示前 4 个字符）
- **FR-022**: 系统 MUST 将配置文件权限设置为仅所有者可读写（Unix 权限 0600）
- **FR-023**: 系统 MUST 在文件权限设置失败时输出警告但继续正常保存配置

#### 用户体验 (FR-024 至 FR-029)

- **FR-024**: 系统 MUST 使用绿色文本显示成功操作信息
- **FR-025**: 系统 MUST 使用红色文本显示错误信息
- **FR-026**: 系统 MUST 使用黄色文本显示警告信息
- **FR-027**: 系统 MUST 使用蓝色文本显示信息性提示
- **FR-028**: 系统 MUST 在所有错误信息前添加 `✗` 前缀
- **FR-029**: 系统 MUST 在所有成功信息前添加 `✓` 前缀

### 关键实体

#### ModelConfig（模型配置）

代表一个模型提供商的完整配置信息：
- **name**: 唯一标识符，用于引用该配置
- **base_url**: API 基础地址
- **api_key**: 认证密钥（敏感信息）
- **model_id**: 模型标识符
- **extra_params**: 额外的键值对参数（可选）

#### AppConfig（应用配置）

代表应用的完整配置状态：
- **models**: 所有模型配置的列表
- **active_models**: 从 Agent 名称到模型名称的映射，表示每个 Agent 当前使用的模型

## 成功标准

### 可衡量的结果

- **SC-001**: 开发者可以在首次运行任何 `asw` 命令时自动完成配置初始化（无需手动执行初始化命令）
- **SC-002**: 开发者可以在 1 分钟内添加并验证一个新的模型配置（从执行 `asw model add` 到 `asw model list` 确认）
- **SC-003**: 配置文件的读写往返操作 100% 保证数据一致性（所有字段值与原始对象完全一致）
- **SC-004**: 所有 API Key 在命令行输出中 100% 被掩码处理（仅显示前 4 个字符）
- **SC-005**: 所有配置文件在创建或修改后权限为 0600（仅所有者可读写）
- **SC-006**: 95% 的错误场景（如权限不足、无效 URL、配置不存在）能够提供清晰的错误信息和解决建议
- **SC-007**: 所有成功操作以绿色文本显示，所有错误以红色文本显示，提供 100% 的视觉一致性

## 假设与依赖

### 假设

1. 用户在类 Unix 系统（Linux、macOS）上运行该工具
2. 用户具有 `~/.agentswitch/` 目录的读写权限
3. 用户熟悉命令行工具的基本操作
4. TOML 格式满足配置存储的需求（不需要复杂的嵌套或引用）

### 依赖

1. Rust 2024 Edition 标准库
2. TOML 序列化/反序列化库
3. 命令行参数解析库（clap）
4. 终端彩色输出库

## 范围外

以下功能明确不在本阶段范围：

- 手动执行的 `asw init` 命令（配置初始化完全自动化）
- 与 Agent 工具（Claude Code、Codex 等）的实际集成和配置文件修改
- 配置预设（Presets）功能
- 配置导入/导出功能
- 配置文件版本控制和迁移
- 多平台支持（Windows、macOS）的差异化处理
- API Key 的加密存储（仅做文件权限保护）
- Web Dashboard 或图形界面
- 配置验证和测试连接功能
